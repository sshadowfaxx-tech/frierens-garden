# 2026-03-24 — ShadowHunter Tracker Fixes (Long-Running Stability)

## Issue Reported
User shared tracker logs after 32,000+ cycles showing:
- `SSLCertVerificationError` on `api.mainnet-beta.solana.com`
- "Too many connections" errors (connection pool exhaustion)
- Errors occurring at cycles 32076-32104+

## Root Causes Identified
1. **Session refresh not applying SSL context** — When refreshing session every 5000 cycles, the new session didn't inherit the SSL context needed for Windows
2. **Connection pool exhaustion** — TCP connections accumulating over long runs without proper cleanup
3. **No fallback on SSL errors** — Single RPC failure would skip wallet for that cycle

## Fixes Applied to trackerv2_clean.py

### 1. Added `_create_session()` Helper Method
```python
def _create_session(self) -> aiohttp.ClientSession:
    """Create aiohttp session with proper SSL context for Windows"""
    connector = aiohttp.TCPConnector(
        limit=50,
        limit_per_host=20,
        ssl=ssl_context if ssl_context else None,
        enable_cleanup_closed=True,  # Clean up closed connections
        force_close=True             # Force close after use
    )
    return aiohttp.ClientSession(
        timeout=aiohttp.ClientTimeout(total=Config.RPC_TIMEOUT, connect=2),
        connector=connector
    )
```

### 2. Updated Session Initialization
Changed from inline session creation to using `_create_session()` to ensure consistent SSL context application.

### 3. Fixed Session Refresh Logic
Session refresh every 5000 cycles now properly applies SSL context via `_create_session()`.

### 4. Added RPC Retry Logic in `check_wallet_fast()`
- Retries up to 3 times (once per available RPC)
- Catches specific exceptions: `asyncio.TimeoutError`, `aiohttp.ClientSSLError`, `aiohttp.ClientConnectorError`
- Automatically rotates to next RPC on failure
- Only logs as error after all retries exhausted

### 5. Improved Exception Handling
Separated exception types for better debugging and graceful degradation.

## Key Configuration Changes
- `enable_cleanup_closed=True` — Prevents stale connections accumulating
- `force_close=True` — Ensures connections are properly closed
- Retry loop with RPC rotation — Handles transient SSL/connection failures

## Critical Fix: "Too many connections" Error (March 24, 23:42)

### Issue Reported
User reported frequent "Too many connections" errors after tracker had been running for ~3,600 cycles. Log showed:
```
05:24:47 | ERROR    | Check error for mW4PZB45: Too many connections
05:24:47 | ERROR    | Check error for 8deJ9xeU: Too many connections
```

### Root Cause
The tracker was spawning tasks for ALL wallets simultaneously (40+ wallets), each competing for DB connections:
```python
tasks = [self.check_wallet_fast(w) for w in self.wallets]  # 40 concurrent tasks!
```

With only 10 DB connections in the pool and 40 concurrent wallet checks, this exhausted PostgreSQL's connection limit.

### Fixes Applied

#### 1. Added `wallet_check_semaphore` (limit: 8 concurrent)
```python
self.wallet_check_semaphore = asyncio.Semaphore(8)
```

#### 2. Throttled Main Loop
```python
async def check_with_semaphore(wallet):
    async with self.wallet_check_semaphore:
        return await self.check_wallet_fast(wallet)

tasks = [check_with_semaphore(w) for w in self.wallets]
```

#### 3. Reduced DB Semaphore (10 → 5)
```python
self.db_semaphore = asyncio.Semaphore(5)
```

#### 4. Reduced DB Pool Size (10 → 8)
```python
max_size=8  # Was 10
```

### Files Modified
- `/root/.openclaw/workspace/trackerv2_clean.py`
- `/root/.openclaw/workspace/increase_pg_connections.py` (new - run as Administrator to set PostgreSQL max_connections to 200)

### Trade-offs
- **Before**: All wallets checked simultaneously (fast but unstable)
- **After**: Max 8 concurrent checks (slightly slower cycles, but stable)

### Deployment Steps
1. Stop tracker: `Ctrl+C`
2. Copy new files to FX 6300
3. Run `increase_pg_connections.py` as Administrator (or manually set `max_connections = 200` in PostgreSQL config)
4. Restart tracker
